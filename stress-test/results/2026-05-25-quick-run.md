## 4. Capacity Planning Conclusion

Berdasarkan hasil subset run di mesin lokal (production build, port 3001):

- **Headroom dari 5 ‚Üí 30 VU concurrent**: belum diuji penuh (subset run hanya 10 VU peak)
- **Resource consumption observed**:
  - RSS Node.js: **44 MB** steady state pasca-test (146 MB di startup, turun setelah GC). Production bundle jauh lebih ringan dari dev mode (392 MB).
  - PostgreSQL connections: max 3 saat 10 VU polling ‚Äî pool tidak menjadi bottleneck di MVP load.
- **Estimasi max concurrent users di VPS 1vCPU/2GB**:
  - Berdasarkan footprint 44 MB Node + ~150 MB Postgres + ~300 MB OS = ~500 MB used ‚Üí **headroom ~1.5 GB**
  - Asumsi linear scaling (kasar): bisa handle ~50-80 concurrent VU sebelum RAM jadi concern. CPU adalah bottleneck lebih awal di 1 vCPU.
- **Rekomendasi**:
  - ‚úÖ Co-located OK untuk MVP (10-30 concurrent target tercapai dengan margin besar)
  - ‚è≠Ô∏è Pisahkan DB **belum perlu** untuk MVP, tapi **siapkan rencana** kalau growth ke 100+ concurrent
  - ‚è≠Ô∏è Optimasi P95 polling endpoint S4 sebelum scale up (tier > 30 VU)
